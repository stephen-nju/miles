# 多进程 Session Server：薄 Router + 经 IPC 分发到无网络 Worker

## Goal Description

为 Miles session server 增加一个**可选**的多进程模式，使 CPU 密集的响应处理（主要是对大 R3 `routed_experts` 响应体的 `json.loads`/校验）能够跨 OS 进程并行、逃离 GIL，而不是像当前的 bounded thread pool 那样在单进程的 GIL 上互相争用。

架构为**一个薄 router 进程**前置 **N 个无网络 worker 进程**：

- router 是唯一面向客户端的 HTTP 监听者。每个请求它从 path 提取 `session_id`，选 `worker_idx = stable_hash(session_id) % n_worker`，经 **IPC（不是第二跳 HTTP）**转发给该 worker，`await` 其 IPC 结果，再回写 HTTP 响应。router 不持有任何 session 状态，且永不 `json.loads` chat 响应体。
- 每个 worker 跑自己的 asyncio event loop 与现有 session 逻辑（`SessionRegistry` / `LinearTrajectory` / per-session in-flight gate 全在进程内）。worker 自己发起对上游推理引擎的 HTTP 调用（`backend_url` 不变），并在本地存完整 record（含 R3）。
- session 之间几乎 standalone（无跨 session 共享状态）；sticky-by-hash 保证同一 session 永远回到同一 worker，所以 IPC 只搬本轮请求/响应 payload，绝不搬累积的 session 状态。

`--session-server-workers=1`（默认）保留今天精确的单进程路径（无 router、无 IPC），使该特性完全可选且向后兼容。

本计划基于已定决策（见 `## Pending User Decisions`，全部 RESOLVED）：
- **基线行为（DEC-0）**：从 `9cf2a0384` 切分支，并在新分支上干净地重建 per-session in-flight gate（409）与严格的 upstream 响应校验（502）；不要带入 bounded CPU thread-pool offload（多进程取代它）。等价目标 = `9cf2a0384` 行为加上这两项重建行为。
- **chat 响应中的 R3（DEC-1=B）**：从面向 client 的 chat 响应里剥离 `routed_experts` / `indexer_topk`，使大 R3 blob 每轮不再跨 IPC；R3 留在 worker 的 record 里，仍通过 `GET /sessions/{session_id}` 提供给训练样本重建。
- **Worker crash（DEC-2）**：任一 worker 死亡都让整个 session server 快速失败（loud failure；不做静默分片重启而让活跃 session 变成带错误状态的 200）。
- **v1 成功标准（DEC-4）**：成功 = 一个正确的、逃离 GIL 的多进程架构，在 production-shaped benchmark 上相对单进程与 thread-pool 路径在聚合响应解析吞吐 / 尾延迟上呈现**方向性**改善。无固定数值门槛。delta/增量 R3 协议明确不在范围内（延后）。

分支：从 commit `9cf2a0384` 开始；创建分支 `refactor/multi-process-session-server`；不要在 `refactor/session-server-concurrency` 上构建。

## Acceptance Criteria

遵循 TDD 理念，每条准则包含正向与负向测试以便确定性验证。行为等价测试应参数化，能同时跑在 `--session-server-workers=1`（单进程 adapter 路径）与 `--session-server-workers=N`（router + IPC）两种部署下。

- AC-1：可选与向后兼容。`--session-server-workers` 未设或 `=1` 时，session server 行为与单进程服务器逐字节一致，且不 spawn router/额外 worker 进程。
  - Positive Tests（应 PASS）：
    - workers=1 时，现有 public-HTTP 套件（`tests/fast/router/test_sessions.py`、`tests/fast/router/test_session_race_conditions.py`、`tests/fast/router/test_session_pretokenized_e2e.py`）原样通过。
    - workers=1 时，启动路径恰好产生一个 session-server 进程（无独立 router 进程、无额外 worker）。
  - Negative Tests（应 FAIL）：
    - workers=1 却 spawn 了 router 或任何额外 worker 进程的配置，未通过进程拓扑断言。
    - 任何改变 workers=1 单进程行为（如不同状态码、丢失 header）的改动，未通过原样套件。

- AC-2：create 路径与 sticky 路由一致性。workers=N 时，经 `POST /sessions` 创建的 session 恰好被一个 worker 拥有，且每个 `session_id`-keyed 端点都路由到同一 worker。
  - Positive Tests：
    - `POST /sessions` → `POST /sessions/{id}/v1/chat/completions` → `GET /sessions/{id}` → `DELETE /sessions/{id}` 都作用于同一 worker 的状态；GET 返回该 chat 追加的 record。
    - catch-all `POST/GET/... /sessions/{id}/{path}`（如 `/abort_request`）路由到拥有它的 worker 并代理到 backend，**不做** session 存在性 404 检查（与今天 catch-all 行为一致）。
  - Negative Tests：
    - 对没有任何 worker 拥有的 `session_id` 的 `chat/completions` / `GET` / `DELETE` 返回 404 `SessionNotFoundError`（router 不得把它路由到某个任意 worker 而静默建出新状态）。
    - catch-all 路径对未知 session id 返回 404（错误地把 create/chat 的 404 规则套到代理路由）即为失败。

- AC-3：稳定的 32-hex session id（firm requirement）。session id 保持 32 字符小写 hex 字符串，且路由用与 `PYTHONHASHSEED` 无关的进程稳定哈希。
  - Positive Tests：
    - `len(session_id) == 32` 且匹配 `[0-9a-f]{32}`（保留 `tests/fast/router/test_sessions.py`）。
    - 同一 `session_id` 在不同进程 / 解释器重启间映射到同一 `worker_idx`（稳定哈希）。
  - Negative Tests：
    - 非 32-hex 的 id 未通过格式断言。
    - 基于 Python 内置 `hash()`（每进程加盐）的路由函数，在进程重启后把某 id 重映射到不同 worker，未通过跨进程确定性测试。

- AC-4：session 核心跨传输的行为等价。解耦后的核心 handler 无论经单进程 FastAPI（workers=1）还是经 router+IPC（workers=N）驱动，都产生一致的 session 状态与错误语义。
  - Positive Tests（在 workers=N 下跑 race + pretokenized 套件）：
    - per-session in-flight gate：并发的第二个同 session chat 返回 409 `SessionBusyError` 且**不**触达上游 backend；不同 session 并发运行。
    - closing（404）优先于 busy（409）；每条错误路径（坏 JSON、400 校验、传输 502、tokenization 500、invariant 500、client cancel/disconnect）都释放 slot。
    - 非 200 upstream 透传不记录；非法 200（缺 meta_info / 非数值 logprob / 非 int token id）→ 502 且不提交任何状态。
    - TITO pretokenized 前缀不变量成立；rollback 至多一个 assistant step（`MAX_ASSISTANT_ROLLBACK_STEPS`）；`append_record` 同样裁剪旧 R3 blob；下游 framing header（`content-length`/`transfer-encoding`/`content-encoding`）被剥除。
  - Negative Tests：
    - worker 把 `SessionError` 映射到错误状态（如 busy session 返回 500）未通过 409 测试。
    - router/worker 记录了非 200 upstream 响应、或在非法 200 上提交了状态，即为失败。
    - worker 在任一错误路径未释放 in-flight slot，未通过 slot 释放测试。

- AC-5：worker 并发模型（厘清草稿里的"每个 worker 串行"）。worker 内部，不同 session 的上游推理调用重叠；CPU parse/validate 有界；每个 session 仍恰好一个 in-flight chat。
  - Positive Tests：
    - 同一 worker 拥有的 K 个不同 session 上并发 K 个 chat，每个上游模拟延迟 L，总耗时 ≈ L（重叠），而非 ≈ K·L。
    - 某 worker 上一个 session 的 `DELETE`/`GET` 在另一 session 的 chat 正等待上游调用时仍被服务（IPC 分发不是把其他 session 堵在一个 chat 后面的单一 FIFO）。
  - Negative Tests：
    - 严格串行的 worker 把上述 K 个不同 session 并发 chat 在 ≈ K·L 完成，即为失败。
    - 无界的进程内 parse 并发复现 16-thread GIL 争用延迟曲线，未通过有界 parse 断言。

- AC-6：大响应路径不把瓶颈搬到 router（DEC-1=B）。R3 从 client chat 响应剥离并留在服务端；router 永不解析 chat 响应体；并发下 router 内存有界。
  - Positive Tests：
    - 开启 R3 时，client chat 响应含 assistant message 与小 meta 字段，但**不**含 `routed_experts` / `indexer_topk`；`GET /sessions/{id}` 的 records 对保留的 turns 仍含 R3，`compute_samples_from_openai_records` 原样重建样本（含 routed_experts）。
    - router 进程永不对 chat 响应体调用 `json.loads`；在 production-shaped benchmark 上，router RSS 不随 `concurrency × body_size` 增长。
    - 在 production-shaped benchmark（32 sessions、50 turns、r3_scale=1000）上，workers=N 相对 workers=1 与 thread-pool 路径在聚合响应解析吞吐 / 尾延迟上呈现改善（方向性；依 DEC-4）。
  - Negative Tests：
    - client chat 响应仍包含完整 `routed_experts` blob，未通过 R3 剥离断言。
    - router `json.loads`/重序列化完整响应 dict，或其 RSS 随 `concurrency × R3_size` 线性增长到 OOM 风险，即为失败。
    - 破坏 `GET`-records R3 取回（导致训练样本重建丢失 routed_experts）的改动即为失败。

- AC-7：生命周期、健康与崩溃语义。启动器在一个非 daemon supervisor 下启动 router + N workers；就绪等待每个 worker 的 tokenizer/TITO 初始化；router `/health` 反映 worker 存活；关闭不留孤儿进程；任一 worker 死亡使整个 session server 快速失败（DEC-2）。
  - Positive Tests：
    - 启动阻塞直到所有 worker 报告就绪；`OpenAIEndpointTracer.create()` 的 `/health`（带稳定的 `session_server_instance_id` 形状）对 router 成功。
    - 对 supervisor 发 `SIGTERM` 会终止 router 与所有 worker，无孤儿进程。
    - 杀掉一个 worker 使 session server 快速失败（router 停止服务 / 暴露 unhealthy），而非静默重启一个空 worker 在后续请求里返回新状态。
    - chat 中途真实 client disconnect 不泄漏 worker 的 in-flight slot 或 router 侧 pending reply（与今天"handler 继续并可能提交该 turn"语义一致，完成时释放 slot）。
  - Negative Tests：
    - supervisor 关闭后仍残留孤儿 worker 进程，即为失败。
    - 死分片被静默替换为一个全新空 worker，对此前存在的 session 返回 200，未通过 fail-fast 策略。
    - client disconnect 后泄漏的 in-flight slot 或 pending IPC reply future，即为失败。

## Path Boundaries

### Upper Bound（最大可接受范围）
一个完整的可选多进程 session server：薄 router 进程（唯一 HTTP 监听者）按 `session_id` 的稳定哈希经 IPC 分发到 N 个无网络 asyncio worker；router 拥有的 32-hex session id，create/chat/get/delete/catch-all 路由一致；session handler 从 FastAPI 解耦为传输中立的核心 callable；per-worker 上游 I/O 重叠 + 有界 CPU parse；重建的 per-session in-flight gate（409）与严格 upstream 响应校验（502）；R3 从 client chat 响应剥离、只留 worker record（仍经 GET records 提供）；非 daemon supervisor，含所有 worker 就绪后启动、`/health` 存活、worker 死亡 fail-fast、关闭无孤儿；以及一个 production-shaped benchmark，跨单进程 / thread-pool / 多进程证明并行解析收益。行为等价套件在 workers=1 与 workers=N 下通过。

### Lower Bound（最小可接受范围）
同样的可选多进程服务器，使用对稳定哈希的简单 `% n_worker`（无一致性哈希）、router 拥有的 32-hex id、解耦的核心 handler、对所有 session-keyed 端点的 sticky 路由、重建的 in-flight gate + 严格校验、R3 从 client 响应剥离、workers=1 不变、满足 AC-1..AC-5 与 AC-7，以及至少一个正确性 benchmark 加 AC-6 的方向性性能测量。IPC 传输可用满足 AC-6（router 永不解析 chat body；router RSS 有界；worker 并发不被串行化）的最简机制。

### Allowed Choices
- Can use：用 `hashlib`（如 `blake2b`/`sha256`）做路由的进程稳定哈希；IPC 用 UNIX-domain socket / `socketpair` framed 二进制协议，或 `multiprocessing` `Pipe`/`Queue`，或共享内存——前提是 AC-5 与 AC-6 成立（不串行化 worker、router 永不 `json.loads` chat body、router RSS 有界）；子进程用 `multiprocessing` spawn 上下文；workers=1 路径用一个调用同一解耦核心的薄 FastAPI adapter；保留 `SessionServer.do_proxy` facade 以减少测试 monkeypatch 改动面。
- Cannot use：把 Python 内置 `hash()` 当路由协议（每进程加盐）；在 primary 热路径上加第二跳 loopback HTTP；为 chat 路径复用 `MilesRouter` 的 `httpx` + `json.loads` + `JSONResponse` 代理核心；不做 per-session 单 owner sticky 而分片 session 状态；让 daemon router/worker 自己再 spawn 子进程（daemonic 进程不能有子进程）；用每 worker 单一 FIFO IPC 分发把其他 session 的请求堵在一个 in-flight chat 后面；任何破坏训练样本重建所用 `GET`-records R3 取回的改动。

> 注：HTTP 反代退路（每 worker 一份未改 `session_server`、router 按哈希反代）在此明确**不在范围**，因为它与"无第二跳 HTTP"的 firm 立场冲突。若解耦 FastAPI handler 被证明不可行，停止并重新规划，而不是静默切到该退路。

## Feasibility Hints and Suggestions

> 注：本节仅供参考与理解，是概念性建议，不是强制要求。

### Conceptual Approach
一种可能的分解：

```
[ rollout client / agent ]
        │  HTTP  (POST /sessions, /sessions/{id}/v1/chat/completions, GET, DELETE, /{path})
        ▼
[ thin router process ]  ── 唯一 HTTP 监听者；无 session 状态；永不 json.loads chat body
        │  从 path 取 session_id；worker_idx = blake2b(session_id) % n_worker
        │  IPC：发送 (request_id, method, path, query, headers, body)；await 回复
        ▼
[ worker_i process ]  ── 自己的 asyncio loop；进程内 SessionRegistry/LinearTrajectory/in-flight gate
        │  核心 handler（传输中立）：占用 per-session slot → 锁下 TITO prepare
        │  → 对 backend_url 的上游 HTTP 调用（跨 session 重叠）→ 校验 → 提交 record（完整 R3）
        │  → 经 IPC 回复：status、headers、body（此 client 面向 body 已剥离 R3）
        ▼
   GET /sessions/{id}  → 返回含 R3 的 records（每 session 一次，在收集时）
```

- `POST /sessions`：由 **router** 生成 32-hex id、算出 owner、并向该 worker 发显式"用此 id 创建"（`SessionRegistry.create_session_with_id(session_id)`），使 create 与后续请求路由一致。
- 把 `setup_session_routes` 闭包解耦为传输中立核心函数，入参 `(method, path, query, headers, body)`，返回 typed result `(status_code, headers, body_bytes, raw_passthrough_flag)`；worker 映射 `SessionError`→status；非 200 透传不记录；router 只写 HTTP 响应。
- worker 并发：保留 per-session in-flight gate；让不同 session 的上游 `await` 在 worker loop 上重叠；用进程本地 semaphore（大小 ~1–2）限制 CPU parse/validate，而非重新制造 GIL 争用的 thread pool。
- IPC 分发必须支持按 `request_id` 的乱序回复，使 `DELETE`/`GET` 不被同 worker 上一个 in-flight chat 堵住。
- workers=1 短路：保留一个包裹同一核心的薄 FastAPI adapter，使现有 HTTP 测试原样跑、且不创建 router/IPC。

## Dependencies and Sequence

### Milestones
1. Milestone M0 —— 基线 + 行为重建：从 `9cf2a0384` 切分支 `refactor/multi-process-session-server`。在该分支上干净地重建 per-session in-flight gate（409）与严格 upstream 响应校验（502/UpstreamResponseError）；不要加 bounded CPU thread-pool offload。这确立了 AC-4 引用的单进程等价目标。
   - Phase A：从 `9cf2a0384` 切分支；确认基线 session 行为（无 gate / 无严格校验 / 已有 raw 透传）。
   - Phase B：重新加入 per-session 单 in-flight 语义 + 严格校验（无 thread pool）；更新/移植对应测试作为等价基线。
2. Milestone M1 —— 把 session 核心从 FastAPI 解耦：抽出传输中立核心 handler（create/get/delete/chat/proxy），入参 `(method, path, query, headers, body)` → typed result，保留 query 转发与请求/响应 header 剥离。保留薄 FastAPI adapter 使 workers=1（及所有现有 HTTP 测试）原样工作。
   - Phase A：引入核心 handler 模块 + typed result；在 FastAPI 之外映射 `SessionError`→status。
   - Phase B：将 `setup_session_routes` / `SessionServer` 重接到经 adapter 调用核心；保留 `do_proxy` facade。
3. Milestone M2 —— worker-count 参数 + router 拥有的稳定 id + create 路由：加 `--session-server-workers`（默认 1 → 单进程路径）。加 `SessionRegistry.create_session_with_id`（校验 32-hex、不覆盖、防碰撞）与 router 侧稳定哈希 id 生成，使 `POST /sessions` 路由一致。
4. Milestone M3 —— router + IPC + 无网络 worker + sticky 路由：构建薄 router 进程与 IPC 通道（乱序、按 request-id、背压、取消/清理）。spawn N 个无网络 worker（无 HTTP 监听）。对所有 session-keyed 端点 sticky 路由；跨 IPC 携带 `SessionError`/status 与原始非 200 透传而 router 不解析。从 client chat 响应剥离 R3（DEC-1=B），worker record 保留完整 R3。
   - 依赖 M1（核心 handler）与 M2（参数 + id + create 路由）。
5. Milestone M4 —— 并发、生命周期、崩溃策略：per-worker 上游 I/O 重叠 + 有界 parse（AC-5）。非 daemon supervisor 拓扑（router + N workers）、所有 worker 就绪后启动、`/health` 存活汇总、关闭无孤儿、worker 死亡 fail-fast（DEC-2）。保留 client-disconnect 语义且不泄漏 slot/reply。
   - 依赖 M3。
6. Milestone M5 —— 等价 + benchmark + 性能验证：把 public-HTTP 套件跨 workers=1/N 参数化（AC-4）。扩展 benchmark 对比 `9cf2a0384` 单进程、thread-pool 路径与多进程；验证方向性并行解析收益与有界 router RSS（AC-6、DEC-4）。
   - 依赖 M3 与 M4。

依赖小结：M0 → M1 → M2 → M3 → {M4, M5}；M5 的生命周期/并发行为基准还依赖 M4。

## Claude-Codex Deliberation

### Agreements
- 多进程需要 per-session sticky 单 owner 路由，因为 `SessionRegistry` 是进程内 dict、`LinearTrajectory.lock`/`chat_inflight` 是进程内状态（代码自己的 "sticky ownership deferred" 注释证实）。
- router 拥有的、稳定的 32-hex session id；worker 必须支持"用 id 创建"。
- worker 必须自己发起上游推理 HTTP 调用；router 永不 `json.loads` chat 响应体（否则瓶颈只是搬到 router）。
- workers=1 必须短路到今天的单进程路径；可选、零行为风险。
- 把 FastAPI 闭包解耦为传输中立核心 handler 是正确的第一步结构改动；v1 用（对稳定哈希的）简单取模即可，一致性哈希延后。
- worker 不可跨 session 严格串行——只串行 per-session 变更并限 CPU parse；让不同 session 的上游 I/O 重叠。

### Resolved Disagreements
- 基线内容（高影响）：Codex 指出——git 也确认——`9cf2a0384` 不含 in-flight gate / 严格校验 / CPU executor（它们在被放弃的分支上、位于 `9cf2a0384` 之上）。解决：加 M0 在新分支干净重建 gate + 严格校验（弃 thread pool），依用户决策 DEC-0。AC-4 的等价目标是此重建行为集，而非当前 HEAD。
- "行为一致" vs R3 剥离：协调一致——等价覆盖 session 状态、错误/gate/校验语义、训练样本重建；面向 client 的 chat 响应有意不同（剥离 R3，DEC-1=B），是唯一记录在案的行为变更，伴随一条测试断言更新。
- catch-all 404：纠正——`/sessions/{id}/{path}` 不得对未知 id 返回 404；它按 session_id sticky 路由并代理 `{path}`（保留 `/abort_request` 行为）。新增 AC-2 负向测试。
- 进程拓扑：纠正——今天启动是 `daemon=True`（fork）；daemon 不能 spawn 子进程，故设计用非 daemon supervisor 持有 router + N worker PID，含显式启动/SIGTERM/join/无孤儿处理（AC-7）。
- pre-body 409 fast-fail：放宽为真正契约——并发同 session chat 返回 409 且**不**进 backend；"读 body 前拒绝"的微优化是允许的实现细节，非硬性要求。
- HTTP 反代退路（Alt-1）：从范围内 allowed choices 移除（与 firm 的"无第二跳 HTTP"冲突）；若 handler 解耦不可行，停止并重新规划。

### Convergence Status
- Final Status: `converged`
- 轮次：Codex 首轮分析（1）+ 一轮收敛复审（1）。Codex 的全部 REQUIRED_CHANGES 均被接受并并入；唯一剩余的对立意见为用户决策，现已全部解决（DEC-0..DEC-5）。无遗留的 Claude/Codex 争议。

# Session Server 多进程化：薄 Router 经 IPC 分发到独立 Worker

## Original Idea

当前分支做了很多 session server 里面做多线程/多进程的努力。但是是徒劳的，没用，一直卡在 gil，没什么优化空间。每轮只返回增量 r3 需要 further discussion，目前做不了。现在有个 idea 是，可以做整个 session server 级别的多进程，举个例子：
1. 我们吧 session server 拆分成 router 和 worker。我们预先启动多个 worker 进程，比如 n_worker。每当有新的 session id 进来时，就走 router，把请求路由到这个 session id 取模 n_worker 上面，用它来实际处理和转发请求。然后每个 worker 内部，依然是串行的。
这么做成立的主要原因是，session id 之间是几乎 standalone 的，所以不用担心内存在进程间 transfer 开销的问题。
2. 你从 9cf2a0384 为基准开始做。不要再 refactor/session-server-concurrency 上面改。开个新分支叫 refactor/multi-process-session-server

## Primary Direction: 薄 Router + 进程间 IPC 分发（router 独占网络）

### Rationale

router 极薄——只做"终止客户端 HTTP + 按 session_id 取模选 worker + 进程间转发"，worker 无网络监听、是纯处理进程；router→worker 走 IPC 而非第二跳 HTTP，这与"HTTP 反代"形态是根本区别，也避开了在 router 处对 R3-heavy 响应再拷贝/再解析的浪费。

### Approach Summary

把当前单进程 session server 拆成"一个薄 router + N 个无网络 worker"，唯一新增的跨进程跳是 IPC，不是 loopback HTTP：

- **Router 进程（薄，async）**：唯一面向 rollout 客户端的 HTTP 监听者。收到 `/sessions/{session_id}/...` 后只做四件事——读取 body、从 path 提取 `session_id`、`worker_idx = hash(session_id) % n_worker`、把 `(method, path, body, headers)` 经 IPC 通道发给 worker_idx 并 `await` 其结果，最后回写 HTTP 响应。router 不持有任何 session 状态、不解析响应体、不再发起第二跳 loopback HTTP。
- **N 个 Worker 进程（无网络监听，各跑自己的 asyncio loop）**：经 IPC 收到被分发的请求 → 执行现有 session 处理逻辑（`SessionRegistry` / `LinearTrajectory` / per-session in-flight gate 全部保留在 worker 进程内）→ 其中对**上游推理引擎**（sglang / miles 推理 router）的调用仍是真实 HTTP（`SessionServer.do_proxy` 的 `backend_url`，保持不变）→ 把响应经 IPC 回传给 router。
- **为何 IPC 而非 HTTP 是根本区别（两个 router 目标不同，不可混为一谈）**：miles 推理 router 的目标是把请求分发到分布在多 GPU / 多节点上的 SGLang 引擎，target 天然可能跨机，所以**必须**走 HTTP——HTTP 对它是正确选择；而 session-server router 的目标完全不同，是把请求分发到**同机的 sibling worker 进程**（session 之间 standalone、都在一台 host 上），因此应走 IPC。在 HTTP 反代形态里，含 R3 blob 的大响应还要在 router 处被完整缓冲、甚至 `json.loads` 再重建一遍（见 Objective Evidence 对 `router.py` 的实证）；改成 IPC 后，router 退化为字节搬运 + 哈希选路，热路径上不再有多余的 JSON 解析与 HTTP 往返。
- **与原 idea 论证一致**：session 之间 standalone，sticky-by-hash 保证同一 session 永远回到同一 worker，所以 IPC 只搬"这一次请求/响应"的 payload，不搬 session 累积状态——不存在进程间 transfer 累积状态的开销。
- 改动落点：新建薄 router；把 session handler 从 FastAPI 路由解耦为可由 IPC 消息直接驱动的纯 async 函数（见 Known Risks）；`miles/ray/rollout/router_manager.py` 的 `start_session_server` 改为 spawn 1 个 router + N 个 headless worker 并建立 IPC 通道；新增类似 `--n-session-server-workers` 的参数。

### Objective Evidence

- **反证"为何不用 HTTP 反代 / 不复用 MilesRouter"**：`miles/router/router.py:130-158` 的 `MilesRouter.do_proxy()` 用 `httpx` 向 worker URL 发起完整 HTTP 请求并 `await response.aread()` 缓冲整个响应体；`build_proxy_response()`（L160-171）进一步对响应体做 `json.loads(content)` 再重建 `JSONResponse`——即把含 R3 blob 的大响应在 router 处完整再解析一遍。注意 MilesRouter 之所以是 HTTP，是因为它的**目标不同**：它要把推理请求分发到可能跨 GPU/跨节点的 SGLang 引擎，跨机就必须 HTTP；而 session-server router 的目标是同机 sibling 进程分发，目标不同 → 传输选择也不同。所以 MilesRouter 的 HTTP 设计对它自己正确，但不适合照搬做本 router 的底座。
- `miles/rollout/session/session_server.py:50-86` 的 `do_proxy` 把请求转发到 `backend_url`（推理引擎），不是转发到 worker；多进程下 worker 承载的正是这套 session + do_proxy 逻辑，对上游引擎的 HTTP 调用保持不变。
- `miles/rollout/session/session_server.py:117-120` 注释明确写道 "Single uvicorn worker on purpose… Multi-process needs sticky session ownership and is deferred"——薄 router + hash 分发正是为补上这条 "sticky session ownership"。
- `miles/ray/rollout/router_manager.py` 已用 `multiprocessing.get_context("spawn").Process` 启动 session server 进程；循环 spawn N 个 headless worker 并在父进程建立 IPC 通道是其自然延伸（spawn 已避免 fork 继承线程的问题）。
- `miles/rollout/session/sessions.py:40-109` 的 `_parse_request_body` / `_dump_request_body` / `_parse_and_validate_response` 已是纯函数并在 `backend.cpu_executor` 中离线执行，便于从 FastAPI 请求对象解耦、改由 IPC 消息驱动。
- `miles/rollout/session/session_types.py` 的 `SessionRecord` 是 Pydantic BaseModel、请求/响应均为 dict，payload 跨 IPC 可序列化（请求侧每 turn 约数百字节；响应侧大小取决于 R3，见 Known Risks）。
- `miles/rollout/session/linear_trajectory.py:248-302` —— `SessionRegistry` 是纯进程内 `dict[session_id -> LinearTrajectory]`，session 间无共享状态，天然支持落在各自 worker 进程内。

### Known Risks

- **session handler 与 FastAPI 解耦（IPC 形态的主要内部成本）**：现有处理逻辑挂在 `setup_session_routes(self.app, ...)` 的 FastAPI 路由上。要由 IPC 消息驱动，需把 handler 从 `fastapi.Request` / `starlette.Response` 解耦成可直接以 `(method, path, body, headers)` 调用的纯 async 函数（或在 worker 内跑一个极小的内部 dispatch）。这是薄 router/IPC 形态相对"HTTP 反代"形态多付的改造成本，也是与保底退路（Alt-1）的主要权衡点。
- **IPC 机制选择 = 性能分水岭（需 further discussion，plan 阶段定型）**：响应体可能很大（R3 blob ~1KB/token，`docs/developer/session-server-overhead.md` 实测保留量可达多 GB）。用 `multiprocessing.Queue` / `Pipe` 会对大响应做 pickle/拷贝，可能抵消掉省下的 HTTP 一跳；用共享内存（`multiprocessing.shared_memory` / mmap）或 UNIX domain socket 传字节流更省。这条直接决定方案是否真优于 HTTP 反代——且直接呼应原 idea 的"每轮只返回增量 r3"：若能只回传增量 R3，IPC payload 会显著变小，该方向收益更明确。
- **worker 内并发语义**：原 idea 写"每个 worker 内部依然串行"。若严格逐请求串行，一个 shard 内多个 session 会被一次上游推理调用阻塞、吞吐受限；更可行的是 worker 跑自己的 asyncio loop 让不同 session 的上游 I/O 重叠（in-flight gate 仍 per-session），而 CPU 工作单线程、跨 worker 才并行（这才是真正绕开 GIL 的点）。需在 plan 阶段把"串行"的确切粒度定清楚。
- **背压与队列容量**：IPC 通道需设上限并处理满/超时；per-session in-flight gate 可缓解同一 session 的堆积，但同一 worker 的多个 session 仍可能堆积。
- **生命周期与可观测性**：headless worker 不再有 HTTP `/health`，需改用进程存活/心跳做健康检查；日志按 worker 分层；统一 graceful shutdown 避免孤儿进程。
- **路由一致性 / 资源倍增**：router 与 worker 对 `session_id` 的哈希口径须一致，否则路由到不拥有该 session 的 worker 会返回 404（`SessionNotFoundError`）；每个 worker 各自加载 tokenizer、各自持 `SessionRegistry`，内存随 `n_worker` 线性增长，`session_server_cpu_workers` 语义变为 per-worker。

## Alternative Directions Considered

### Alt-1: HTTP 反代 + 独立 worker 服务（含复用 MilesRouter，作保底退路）
- Gist: 每个 worker 就是一份完整、未改动的 `session_server` 实例，各监听 `127.0.0.1` 独立端口；router 用 `httpx` 按 `hash(session_id) % n_worker` 反向代理转发。可直接拿 `MilesRouter` 当底座，仅把 `_use_url()` 的最小负载选择换成哈希取模。最大优点是 `sessions.py` / `linear_trajectory.py` 内核零改动、最快跑通。（原稿中"复用 MilesRouter"已并入此项，因二者本质同为 HTTP 反代，避免重复。）
- Objective Evidence:
  - `miles/router/router.py:16-228` 的 `MilesRouter` 已具备 worker 注册、健康检查循环、死 worker 隔离与 `/{path:path}` catch-all 反代，改选择函数即可。
  - `tests/e2e/sglang/test_r3_router_equivalence.py` 与 `tests/fast/router/test_router.py` 已对该 router 的 worker 管理做生产级/单元测试。
- Why not primary: 用户明确指出 router 应极薄、走进程间转发而非 HTTP 转发。该形态多一跳 loopback HTTP，且 router 会把含 R3 blob 的大响应缓冲并 `json.loads` 再重建（`router.py:160-171` 实证），热路径开销大。仅作"若 handler 解耦成本过高"时的保底退路与可行性验证起点。

### Alt-2: SO_REUSEPORT / fd 传递预 fork 进程池
- Gist: 预 fork `n_worker` 个进程共享同一监听 socket（`SO_REUSEPORT` 或 fd 继承），靠 OS 级负载均衡省掉 router 转发，再在其上叠加 `session_id` 亲和（router 接受连接后传 fd，或 worker 自行按哈希取舍）。
- Objective Evidence:
  - `miles/utils/http_utils.py:18-40` 已有 `socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)` 的 socket 操作模式；`uvicorn.run()` 支持 `fd` 参数，但仓库现以黑盒 `uvicorn.run()` 启动，未启用。
- Why not primary: OS LB 默认不保证 session 亲和，必须额外补一层亲和路由；改动深入 uvicorn/socket 启动方式、可移植性差，性价比低（探索自评 low）。与 primary 相比，它连"请求级转发"都省了（直接连接级），但亲和与生命周期更难控。

### Alt-3: 一致性哈希可重平衡路由
- Gist: 把路由/亲和策略做成一等组件——带虚拟节点的一致性哈希环替代简单取模，支持 `n_worker` 动态增减时仅重映射约 `1/n_worker` 的 session、热点感知与 graceful drain，与传输层（IPC/HTTP）正交。
- Objective Evidence:
  - `miles/backends/sglang_utils/arguments.py:22-26` 已支持 `--sglang-router-policy consistent_hashing`；`miles/rollout/sglang_rollout.py:195-198` 用 `X-SMG-Routing-Key` header 传 `session_id`，证明本生态已落地一致性哈希。
  - 当前 `miles/router/router.py` 仅有最小负载选择；`miles/utils/arguments.py` 尚无 `n_session_server_workers` 类参数。
- Why not primary: 它解决的是"扩缩容/热点"进阶问题而非首版可行性，更适合作为 primary 路由策略的可插拔升级（首版用简单取模）。

### Alt-4: 无状态 worker + 外置 session 存储
- Gist: 反转 pin 前提——把 session 状态外置到共享存储（shm / Redis / 文件），worker 无状态、任意 worker 服务任意 session，router 自由均衡且天然容错；每轮 load → 处理 → save。
- Objective Evidence:
  - `docs/developer/session-server-overhead.md` 量化：主导开销是 response body JSON 解析（16-worker 下 parse p50 ≈ 3.36s），而非状态共享；外置存储的 serialize/deserialize 会**放大**该开销。
  - 同文档：未裁剪时 R3 blob 可达约 78 GB（裁剪后约 6 GB），说明 session state 体量大、跨进程搬运成本高；仓库内无分布式 session 存储 / 跨进程 CAS 的现成实现。
- Why not primary: 直接否定了"session standalone 所以 pin 便宜"的收益前提——既不解决 GIL，又新增序列化/分布式锁成本，还要从零设计一致性协议，风险与工作量最高（探索自评 low）。主要价值是前置告警（见 Synthesis Notes）。

## Synthesis Notes

primary 选定后，几条仍值得带进 plan 阶段的取舍：(1) **Alt-1 是 primary 的保底退路**——若把 session handler 从 FastAPI 解耦的成本过高，可退回"每 worker 跑整份现有 `session_server` + router 用 `httpx` 反代（甚至直接复用 `MilesRouter` 改选择函数）"，代价是多一跳 HTTP + 大响应再 parse，但内核零改动、能最快跑通；建议先用它做可行性验证，再切到 IPC 省热路径。(2) **MilesRouter 仅剩 bookkeeping 思路可借**——其 worker 注册表/健康检查/死 worker 隔离的结构可参考，但 proxy 核心（`httpx` + `json.loads` 重建）对 primary 不适用、不应复用；primary 的 router 是全新的薄进程。(3) **IPC 机制是 primary 成败关键**——应在 plan 阶段先用 `docs/developer/session-server-overhead.md` 的基准量化"大响应跨 IPC 的拷贝/序列化成本"，再在 `Queue`/`Pipe` vs 共享内存 vs UDS 间定型；这也与原 idea 的"每轮只返回增量 r3 需要 further discussion"强相关——若能只回传增量 R3，IPC payload 大幅缩小，primary 收益最明确，因此"增量 R3"值得作为并行子课题推进。(4) **Alt-3 一致性哈希**作为 primary 路由策略的可插拔升级，首版简单取模、出现扩缩容/热点长尾时再启用。(5) **Alt-4** 作为前置告警：投入多进程前先用基准确认瓶颈确实是 GIL/串行，而非 response JSON 解析本身，否则任何进程级方案收益都会被高估。
