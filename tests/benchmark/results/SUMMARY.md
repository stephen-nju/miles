# Session-server overhead: OLD (single-process) vs NEW (multi-process)

Generated by `tests/benchmark/bench_session_server_overhead.py` (`--mode http`) on a
~192-core box, mock R3 backend (no GPU/SGLang). Production-shaped workload:
`sessions=32 turns=50 input-tokens=1000 output-tokens=1000 r3-scale=1000`
(final-turn response body ~= 134 MiB; ~78 GiB total R3 transferred per run).

"reply latency" = end-to-end per-chat latency over HTTP (includes queueing).
"peak RSS" = peak summed RSS of the server process group only (single proc for
OLD; router + N workers for NEW), excluding the bench driver.

## Headline (create -> 50x chat -> delete; no full-records GET)

| run | wall (s) | throughput (turns/s) | reply p50 (ms) | reply p95 (ms) | reply p99 (ms) | peak RSS (GiB) |
|---|---:|---:|---:|---:|---:|---:|
| OLD single-process (w1) | 405.3 | 3.95 | 7762.7 | 16223.8 | 18476.6 | 43.6 |
| NEW multi-process (w16) | 60.7 | 26.37 | 750.3 | 1819.3 | 2261.9 | 44.9 |
| NEW multi-process (w32) | 71.8 | 22.27 | 712.1 | 2137.9 | 2513.8 | 56.3 |

Speedup NEW w16 vs OLD w1: wall 6.7x, throughput 6.7x, reply p50 10.3x, p95 8.9x, p99 8.2x.

CPU floor (serial in-process micro-bench, no HTTP): wall 145.8 s, 11.0 turns/s,
response parse+validate p50/p95/p99 = 79.1/160.2/172.6 ms -- confirms the
dominant per-turn cost is the large-body json.loads/validate, matching the
session-server-overhead design note.

## With --get-records (adds a ~190 MiB retained-records GET per session)

| run | wall (s) | throughput (turns/s) | reply p99 (ms) | GET-records p50/p95/p99 (ms) | peak RSS (GiB) |
|---|---:|---:|---:|---:|---:|
| OLD w1 +get-records | 473.7 | 3.38 | 44469.4 | 39249 / 53983 / 54009 | 52.2 |
| NEW w32 +get-records | 63.3 | 25.26 | 1898.5 | 8063 / 15035 / 15469 | 60.8 |

NEW cuts the big-GET p99 ~3.5x and overall reply p99 ~23x.

## Verdict

The multi-process path is a significant, unambiguous win for the
production-shaped large-R3 workload: ~6-7x wall/throughput and ~8-10x lower reply
latency across p50/p95/p99. Root cause is the design thesis -- OLD serializes 32
concurrent ~134 MiB json.loads on one event loop under one GIL; NEW spreads them
across N worker processes (N GILs) behind a thin router. Peak RSS is roughly flat
at w16 (router + 16 workers ~= single-process retention) and grows at w32.

w16 is the sweet spot for 32 sessions: w32 is slightly slower and heavier (worker
parse_concurrency=2 x 16 workers already gives 32 parse slots = the 32 concurrent
sessions, so w32 over-subscribes and adds tokenizer-copy memory).

## Stability

All NEW runs completed 1600/1600 chats with zero worker deaths / fail-fast /
cross-session 503s. The earlier "worker dies ~248 turns" report was a
benchmark-harness keep-alive connection-reset cascade tripping the supervisor's
by-design fail-fast -- NOT a worker/IPC/supervisor defect (root-caused with
opt-in worker crash-capture across 6 full runs incl. original + harsher
conditions: zero captured worker tracebacks). The invariant is pinned by
tests/fast/router/test_session_multiprocess_disconnect.py.

## Caveats

- Mock backend returns synthetic large-R3 bodies; this isolates session-server
  CPU/IPC overhead, not end-to-end model latency.
- Workload-sensitive: the gain scales with concurrency x body size; for small
  bodies or low concurrency it shrinks.

Raw per-run JSON artifacts live beside this file (*.json); the bulky uvicorn
access logs (*.log) are git-ignored.
